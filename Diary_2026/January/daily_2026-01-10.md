# Modem 配置丢失问题分析报告

  

## 问题现象

恢复默认后，配置 CLI 参数（如 `network-mode 4G`）并执行 `write` 操作，之前的 modem 配置（如 `freqBand`、`freqArfcn` 等）丢失。

  

## 根本原因分析



### 1. 核心问题：恢复默认后，内存中的 `tmp_modem_if` 未从新的配置文件加载

  

**对比 ipsec 和 modem 的实现差异：**

  

| 对比项 | ipsec | modem | 问题 |

|--------|-------|-------|------|

| **接口创建时的行为** | 如果已存在，直接返回；如果不存在，创建新接口（calloc，字段为 NULL） | **相同** | ✅ 一致 |

| **恢复默认后的处理** | 配置文件被替换，但**内存链表可能保留旧数据** | **相同** | ⚠️ 都存在风险 |

| **conf_write 逻辑** | `if (!modify) break;` - 遇到第一个 `modify=FALSE` 就停止 | `if (!modified) continue;` - 跳过 `modified=FALSE` 的接口 | ⚠️ 差异可能影响行为 |

| **配置加载时机** | **只在 `ipsec` 进程的 `main` 函数中从配置文件读取（用于生成 ipsec.conf）**，**CLI 内存结构不预加载** | **没有任何预加载逻辑** | ❌ **关键问题** |

  

### 2. 关键代码位置

  

#### ipsec 的实现（`app/cli/append/src/ipsec_conf.c:331-334`）

```c

if (tmp_ipsec_if)

{

    g_ipsec_t = tmp_ipsec_if;

    return CLI_SUCCESS;  // ★ 直接返回，不加载配置

}

```

  

#### modem 的实现（`app/cli/append/src/modem_conf.c:434-440`）

```c

if (tmp_modem_if)

{

    msg(M_INFO, "[DEBUG] config_interface_modem_ppp: ✓ modem %s already exists in memory...");

    vty_out(vty, "the modem-ppp already exist, now edit it%s", VTY_NEWLINE);

    g_modem_if = tmp_modem_if;  // ★ 直接使用内存中的接口，不加载配置

    return 0;

}

```

  

### 3. 问题流程分析

  

#### 场景：恢复默认后配置 modem

  

1. **恢复默认操作** (`app/syscfg/syscfg.c:79-108`)

   ```c

   system("rm -rf /etc/hdconfig/cli.conf");

   system("cp /etc/defconfig/cli.conf /tmp/hdconfig/cli.conf");

   system("killall cli");

   ```

   - 配置文件被替换为默认配置

   - CLI 进程被杀死并重启

   - **重启后，`g_modem_set.modem_t` 链表应该被初始化为 `{NULL, 0}`（空链表）**

  

2. **用户进入配置模式** (`interface modem sim1.1`)

   - `config_interface_modem_ppp_cmd` 被调用

   - `find_modem_if("sim1.1")` 查找内存中的接口

   - **如果接口不存在，创建新接口**（`calloc`，所有字段为 `NULL`）

   - **关键：新创建的接口字段都是 NULL，没有从配置文件加载！**

  

3. **用户配置参数** (`network-mode 4G`)

   - `config_ppp_net_mode` 被调用

   - 只有 `netmode` 字段被设置，其他字段（`freqBand`、`freqArfcn` 等）仍然是 `NULL`

   - `modified = TRUE`

  

4. **执行 write 操作** (`modem_rule_conf_write`)

   - 遍历 `g_modem_set.modem_t` 链表

   - 只写出 `modified=TRUE` 的接口

   - 对于每个字段，**只有非 NULL 的字段才会被写出**：

     ```c

     if (tmp_modem_if->freqBand)

     {

         vty_out(vty, " freqBand %s%s", tmp_modem_if->freqBand, VTY_NEWLINE);

     }

     // ★ 如果 freqBand 是 NULL，这行不会执行，配置丢失！

     ```

   - **结果：只有 `network-mode 4G` 被写出，其他字段丢失**

  

### 4. 为什么 ipsec 不会丢配置？

  

**ipsec 也不会预加载配置到内存，但是：**

- ipsec 的 `conf_write` 使用 `break` 而不是 `continue`，意味着遇到第一个 `modify=FALSE` 的接口就停止遍历

- 如果恢复默认后，ipsec 接口的内存结构被清空，则不会写出任何内容

- **或者，ipsec 的使用场景不同，不会出现恢复默认后立即配置的情况**

  

### 5. 修复方案对比

  

#### 方案1：在接口创建时从配置文件加载（推荐）

**位置**：`config_interface_modem_ppp_cmd` 函数中，当创建新接口时

  

```c

if (!tmp_modem_if)  // 接口不存在

{

    g_modem_if = (modem_if_t *)calloc(1, sizeof(modem_if_t));

    // ... 设置 modem_name ...

    LIST_ADD(&g_modem_set.modem_t, &g_modem_if->entry);

    // ★ 关键修复：从配置文件加载完整配置

    struct modem_paremeter_struct modem_para;

    char modem_name_full[128];

    snprintf(modem_name_full, sizeof(modem_name_full), "interface modem %s", argv[0]);

    memset(&modem_para, 0, sizeof(modem_para));

    init_modem_description(&modem_para);

    modem_para.module_description = modem_name_full;

    if (G3_SUCCEED == cli_config_get((struct paremeter_struct *)&modem_para,

                                      sizeof(struct modem_paremeter_struct),

                                      SAME_PARE_OFF))

    {

        // 加载所有字段到 g_modem_if

        // ...

        g_modem_if->modified = FALSE;  // 从文件加载的配置，不标记为已修改

    }

}

```

  

#### 方案2：在接口已存在但字段为 NULL 时加载配置

**位置**：`config_interface_modem_ppp_cmd` 函数中，当接口已存在时

  

```c

if (tmp_modem_if)  // 接口已存在

{

    // ★ 检查是否已加载完整配置（通过检查关键字段）

    if (!tmp_modem_if->freqBand && !tmp_modem_if->freqArfcn && !tmp_modem_if->freqPcid)

    {

        // 从配置文件加载完整配置

        // ...

    }

    g_modem_if = tmp_modem_if;

    return 0;

}

```

  

#### 方案3：恢复默认时清空内存链表

**位置**：`device_default()` 或恢复默认相关函数

  

**问题**：恢复默认后，CLI 进程会被重启，链表应该已经清空。如果还有问题，可能是进程重启时机的问题。

  

## 推荐修复方案

  

**采用方案1 + 方案2 的组合**：

  

1. **新接口创建时**：从配置文件加载完整配置

2. **接口已存在时**：检查是否已加载完整配置，如果没有则加载

3. **加载后的行为**：设置 `modified = FALSE`，确保只有用户修改的字段才会被标记为已修改

  

这样可以确保：

- 恢复默认后，新创建的接口会从默认配置文件加载完整配置

- 如果接口已存在但字段不完整，也会从配置文件加载

- 只有用户明确修改的配置才会被写出

  

## 需要验证的问题

  

1. 恢复默认后，CLI 进程是否确实重启？内存链表是否被清空？

2. 如果链表没有被清空，为什么？

3. ipsec 是否真的不会出现这个问题？还是使用场景不同？
